# 每完成一個 Minimum Viable Change 就先提交，並盤點剩餘未提交變更

## 核心要求

- 當這一輪最小可交付變更已落地，且已有對應的最小驗證證據後，應先建立一次本地 commit，再繼續往下做。
- `Minimum Viable Change` 指的是一個可獨立說明、可獨立驗證、可獨立回滾的最小變更單位，而不是整個大需求全部完成後才一起提交。
- 完成第一次 commit 後，應順帶盤點當前仍未提交的變更，嘗試再找出下一個可獨立成立的 Minimum Viable Change。
- 若剩餘變更中還能拆出新的 Minimum Viable Change，應先整理並提交該單位，而不是把多個已成立的變更長時間混在同一工作樹中。
- 若剩餘變更還無法形成新的 Minimum Viable Change，應明確回報未提交部分與下一個 Minimum Viable Change 之間仍待補完的差距。
- 若使用者明確要求不要 commit、延後 commit，或目前工作樹含有不應代為處理的他人變更，應先遵從限制並改為回報狀態。

## 為什麼需要這條規則

agent 與開發者在連續工作時，常會先把第一個可交付修正做完，卻因為「反正等等還要繼續」而不立即提交。
結果容易變成：

- 多個已成立的修改混在同一批未提交變更中
- 後續再拆 commit 時，難以分辨哪部分已經可交付
- 若中途被打斷，只能留下大塊未整理的工作樹
- 代理人無法清楚說明哪些部分已完成、哪些部分還差一步

固定在每個 Minimum Viable Change 完成後先提交，可以讓進度更穩定、回滾更容易，也能逼近「每次中斷時都知道現在差在哪裡」的工作節奏。

## 本地同步時要討論什麼

- 本地專案如何界定 Minimum Viable Change，是否需要搭配特定驗證門檻。
- 本地是否允許 agent 自動 commit，或需要特定情境下才啟用。
- 若工作樹同時包含使用者既有未提交變更，應如何區分 agent 可整理的部分與不可碰的部分。
- 無法再拆出下一個 Minimum Viable Change 時，應把差距記錄在回報、runbook 還是其他固定位置。

## 驗證方式

- 檢查完成一個可獨立驗證的最小變更後，是否有立即建立對應 commit，而不是持續累積在工作樹中。
- 檢查 commit 後是否有主動盤點剩餘未提交變更，並評估能否再拆出下一個 Minimum Viable Change。
- 檢查若無法再拆出下一個 Minimum Viable Change，是否有明確回報未提交部分還缺哪些補完點。
- 檢查使用者明示不要 commit 或工作樹含他人變更時，是否改為回報狀態而非逕自提交。
