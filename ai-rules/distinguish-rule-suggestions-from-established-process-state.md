# 區分新的規則建議與既定流程狀態說明

## 核心要求

- 回報中的「規則建議」區塊只應放這一輪新收斂出的規則改善、加嚴建議，或尚未落地的後續規則項。
- 已存在的基線同步機制、既定 review 流程、既有規則的狀態說明，應視為流程狀態或續作脈絡，不應包裝成新的規則建議。
- 若某段內容只是說明「下次從哪個 baseline 開始」、「這輪 topic 已完成」或其他正常續作方式，應放在一般進度說明，而不是規則建議區塊。
- 只有當流程本身被修改、加嚴、放寬，或新增新的治理邊界時，才應提升為規則建議或 shared rule follow-up topic。

## 為什麼需要這條規則

agent 在停工回報、shared baseline review 或治理 follow-up 中，常會同時提到：

- 這輪新發現、值得固定下來的治理改善
- 已經存在、只是剛被執行或剛完成的既定流程

若兩者沒有清楚分開，常見結果是：

- 既有流程被誤寫成新的規則提案
- 規則建議區塊混入單純狀態更新，讓真正該討論的治理改善被稀釋
- shared rule follow-up topic 充滿其實早已存在的 baseline mechanics，增加不必要的 review 成本

先區分「新規則」與「既定流程結果」，能讓治理討論更聚焦，也避免重複追蹤已完成的 topic。

## 本地同步時要討論什麼

- 本地停工回報、治理 review 或 baseline follow-up 是否已有明確區分「規則建議」與「流程狀態」。
- 哪些內容其實只是既有同步模型、review 起點、baseline 前移時機，不應再次包裝成建議項。
- 若本地確實需要規則建議區塊，是否已有清楚邊界，避免把續作方式或完成狀態塞進去。
- 哪些 shared baseline follow-up topic 其實屬於既定流程結果，應從 topic 清單移除而改寫成一般進度說明。

## 驗證方式

- 檢查規則建議區塊是否只保留新的治理改善、加嚴建議或尚未落地的後續規則項。
- 檢查既定 baseline mechanics、next-review flow 與已完成 topic，是否被寫成一般狀態說明而非新規則建議。
- 檢查 shared baseline review 的 topic 清單，是否聚焦在真正需要本地判斷與採納的差異，而不是重述既有流程。
