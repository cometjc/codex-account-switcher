# 開始新 MVC 前先處理工作樹：已存在之 MVC 與不明變更

## 與其他章節的分工

- [commit-each-minimum-viable-change.md](commit-each-minimum-viable-change.md)：每一個 MVC **完成後**應盡快 commit，並盤點剩餘變更。
- **本章**：在**著手下一個** MVC 的修改**之前**，工作樹上若仍有未提交變更，應先釐清並收斂，避免與新工作混雜。

## 核心要求

1. **開工前先看 `git status`（與必要時的 `git diff`）**  
   - 若已有可獨立說明、可獨立驗證、可獨立回滾的變更（符合 MVC），應**先**為該 MVC 建立 commit，再開始新的 MVC。  
   - 與 [commit-each-minimum-viable-change.md](commit-each-minimum-viable-change.md) 一致：不要讓多個已成立的 MVC 長時間堆在同一工作樹。

2. **對「本 session 未產出、來源不明」的變更**  
   - 假設其**可能**為另一 agent、另一工作階段或使用者手動編輯的進行中工作。  
   - **不得**在未釐清前與新 MVC 混寫或一併提交。

3. **評鑑不明變更是否已成 MVC**  
   - 閱讀 diff：是否意圖清楚、是否半成品或僅實驗性修改。  
   - 若專案有對應測試或驗證指令，**應執行**並以結果輔助判斷（通過仍不代表語意完整，但失敗通常表示未完工）。  
   - 若可合理認定已構成一個 MVC：使用適當的 Conventional Commit 訊息**單獨 commit**；若**無法**認定完工或語意不清：**不要 commit**，保留工作樹狀態，並向使用者**回報**有哪些路徑仍髒、建議由誰接續或需補何驗證。

4. **使用者明示限制時**  
   - 若使用者要求不要 commit、或指明某變更不可動：遵從並只回報狀態（與 commit-each 章節一致）。

## 驗證方式

- 新的 MVC 實作開始前，能說明當時工作樹：已提交的 MVC、刻意保留未提交的 WIP 及其理由。  
- 對不明變更曾做 diff 檢視；有測試時曾執行並記錄結果或無法執行的原因。  
- 未將「不明且未完成」的變更與新 MVC 捆綁提交。

## 為什麼需要這條規則

多 agent 或分段工作時，工作樹上的髒檔容易混來源。若直接在新 MVC 上疊加，會造成 authorship 不清、難以回滾、以及誤把他人 WIP 當成可交付狀態。開工前先釐清並分離 MVC，可降低這類風險。
