# Spec Directory Rules

- spec 應描述使用者可感知的行為、流程、限制與驗收結果，避免過度綁定實作技術細節。
- spec 只記錄已實作且已驗證的落地行為；如果內容仍在設計、規劃或尚未驗證，應留在 `plan/`，不要提前寫入 `spec/`。
- 優先說明功能目的、操作體驗、UI 呈現、狀態轉換與例外情境，而不是先限定框架、套件、語言或內部模組切分。
- 若需要提及技術背景，僅保留理解需求所必需的最小資訊，避免把暫時性的實作選型寫成長期規格承諾。
- 若 UI/CLI 含有可比較的數值資訊，應將數值拆成可對齊欄位後再格式化，例如 `2%`、`(94%)`、`6.7d`、`2.1h`、`+1.3%`、`-55.4%` 等，避免把多個數值黏成單一不可控字串。
- 這類數值欄位預設採右對齊，讓不同位數與正負號能跨列穩定掃讀；若有描述文字，描述應與數值分欄處理。
- 若 UI/CLI 使用背景色表達程度，色盤應優先選擇 light/dark theme 都可讀的通用色調，必要時明確指定前景色，避免依賴終端預設前景而在某些主題下失去對比。
- 代表風險或 overuse 的背景色應避免過亮或過淺，優先使用較深的警示色階，讓資訊密集畫面下仍保有層次與可讀性。
- 當某個最小 MVP 完成時，應更新或新增對應 spec，反映目前實際交付的功能、UX、UI。
- 若規格同時涵蓋已實作與待實作內容，需清楚區分兩者，避免讀者誤判完成度。
